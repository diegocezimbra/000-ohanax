SEÇÃO 45

O que Diferencia um Dev Senior na Era da IA

Você chegou até aqui. 44 seções cobrindo desde SOLID até AI Agents, de Clean Code a Quality Attributes. Mas ter conhecimento técnico e condição necessária, não suficiente. O que realmente diferencia um desenvolvedor senior não é saber mais — e saber aplicar com discernimento, liderar pelo exemplo e gerar impacto desproporcionalmente maior que o número de linhas de código que escreve.

Na era da IA, isso é ainda mais verdade. Quando qualquer junior com Claude/Copilot pode gerar 500 linhas de código em minutos, o valor do senior não está em digitar rápido. Está em saber o que digitar, saber o que não digitar, e saber por que. Esta seção organiza esse "meta-conhecimento" em 6 dimensões.

Dimensão 1: Profundidade Técnica

Um senior não é "generalista raso". E alguém com profundidade real em pelo menós 1 ecossistema é amplitude suficiente para transitar entre domínios. As 44 seções anteriores cobriram a amplitude. Agora, vamos falar sobre profundidade.

Dominando Patterns e Trade-offs

O junior conhece o pattern. O senior sabe quando NÃO usar. Cada pattern das seções 1-44 tem trade-offs:

  • Microsserviços — Resolvem escalabilidade de time e deploy independente. Mas adicionam complexidade de rede, distributed transactions e observabilidade. Para times < 5 devs, monolito modular e quase sempre melhor
  • Event Sourcing — Auditabilidade perfeita e time-travel de estado. Mas queries de leitura são complexas (precisa de CQRS), rebuild de projeções é lento e eventual consistency confunde usuários
  • Serverless — Zero ops e escala automática. Mas cold starts, vendor lock-in, debugging local é dolorosó e custos imprevisíveis em escala
  • GraphQL — Frontend pede exatamente o que precisa. Mas caching é mais complexo (não basta CDN por URL), N+1 queries no resolver é o raté limiting e não-trivial
i
Exercício de profundidade: Para cada tecnologia que você usa, escreva 3 cenários onde ela é a melhor opção E 3 cenários onde é a pior. Se não consegue, você não tem profundidade suficiente nela.

Expertise Profunda em 1 Ecossistema

Escolha uma stack e va até o fundo. Não e sobre "saber Node.js" — e sobre:

  • Entender o event loop, o libuv, como o garbage collector V8 funciona é quando causa latency spikes
  • Saber diagnosticar memory leaks com heap snapshots, CPU profiling com flame graphs
  • Conhecer os internals do framework (NestJS: como o DI container resolve dependências, lifecycle hooks, middleware pipeline)
  • Ter opinião informada sobre ORMs (TypeORM vs Prisma vs Drizzle) baseada em experiência real com cada um
  • Entender como o runtime interáge com o SO (file descriptors, TCP connections, epoll)

Trade-off Analysis Framework

Para toda decisão técnica, um senior avalia sistematicamente:

Framework de Análise de Trade-offs
  1. Complexidade: Quanto complexidade isso adiciona? Quem vai manter isso daqui a 2 anos?
  2. Reversibilidade: Se essa decisão estiver errada, quao caro e reverter? (Decisões reversíveis = decida rápido. Irreversíveis = análise profundamente)
  3. Escala: Essa solução funciona para 10x a carga atual? E para 100x?
  4. Time context: O time tem experiência com isso? Quanto custa o ramp-up?
  5. Alternativas: Que outras opções considerei? Por que descartei cada uma?

Dimensão 2: Pensamento Arquitetural

O senior não apenas implementa features — ele projeta sistemas. Isso inclui System Design, NFRs, seleção de tecnologia, documentação de decisões e capacity planning.

ADRs — Architecture Decision Records

Toda decisão arquitetural significativa deve ser documentada. ADRs são o "commit history" das decisões técnicas. Template:

# ADR-007: Usar PostgreSQL como banco primário ao invés de MongoDB

## Status: Accepted
## Data: 2025-03-15
## Autores: @joao, @maria

## Contexto
Precisamos escolher o banco principal para o sistema de pedidos.
O sistema tem:
- Relações complexas (pedido → itens → produtos → categorias)
- Necessidade de transações ACID para pagamentos
- Queries analiticas para dashboard

## Decisão
Usar PostgreSQL 16 como banco primário.

## Alternativas Consideradas
1. MongoDB — Descartado: relações complexas ficam desnormalizadas
   e inconsistentes. Transações multi-document são lentas.
2. MySQL — Viável, mas PostgreSQL tem melhor suporte a JSON,
   CTEs, window functions e extensões (pgvector para futuro AI).
3. DynamoDB — Descartado: queries analiticas exigiriam ETL
   para Redshift. Custo operacional maior para nossó caso.

## Consequências
- (+) Transações ACID nativas para pagamentos
- (+) JSON columns para dados semi-estruturados sem sacrificar relacional
- (+) Ecossistema maduro, fácil contratar devs
- (-) Scaling horizontal mais complexo que DynamoDB (mas desnecessário para nossó volume)
- (-) Precisa de expertise em tuning de queries e vacuum

## Métricas de Sucesso
- P99 de queries < 100ms com 10K pedidos/dia
- Zero inconsistências de transação em 6 meses

Capacity Planning — Back-of-Envelope Calculations

Um senior estima números na cabeça (ou no guardanapo) antes de projetar:

Cenário: API de e-commerce para a Black Friday

--- Estimativa de Trafego ---
Usuários ativos diarios (DAU): 500K
Pico na Black Friday: 10x = 5M DAU
Cada usuário faz ~20 requests/sessão
Total requests/dia no pico: 5M * 20 = 100M requests/dia
RPS medio: 100M / 86400 = ~1,150 RPS
RPS pico (3x medio): ~3,500 RPS

--- Estimativa de Storage ---
Cada pedido: ~2KB (JSON comprimido)
Pedidos/dia no pico: 200K
Storage de pedidos/ano: 200K * 365 * 2KB = ~146GB/ano
Com replicas (3x): ~440GB

--- Estimativa de Banda ---
Response medio: 5KB
RPS pico * 5KB = 3,500 * 5KB = 17.5 MB/s = ~140 Mbps
CDN para assets estáticos: separado, ~500 Mbps no pico

--- Decisões derivadas ---
- 3,500 RPS: um único PostgreSQL com read replicas aguenta
- Não precisa de sharding (volume < 1TB)
- CDN obrigatória para assets (Cloudfront)
- Redis para cache de catálogo (hit raté > 90%)
- Fila (SQS) para processamento de pedidos async

Technology Selection

O senior avalia tecnologia como investimento, não como preferência pessoal:

  • Maturidade: Produção em empresas do nossó porte? Quantos anos? Issues abertos no GitHub?
  • Ecossistema: Libraries, ferramentas, integração com nossó stack existente?
  • Talent pool: Conseguimos contratar devs que conhecem isso? Qual o custo?
  • Exit cost: Se precisarmos trocar em 2 anos, quanto custa a migração?
  • Maintenance burden: Quem mantém o projeto? E uma empresa ou 1 dev open-source?

Dimensão 3: Habilidades na Era da IA

A IA não substituiu devs seniors — ela amplificou a diferença entre quem sabe o que está fazendo e quem não sabe. Um junior com IA gera código mediocre 10x mais rápido. Um senior com IA gera código excelente 10x mais rápido. A diferença está em 3 habilidades.

1. Prompting Eficaz para Geração de Código

  • Sejá específico sobre o contexto: "NestJS 11 com TypeORM, PostgreSQL, class-válidator. Padrão repository com DI" gera resultado 10x melhor que "crie um CRUD"
  • Defina constraints: "Sem usar any. Strict TypeScript. Handlers devem retornar DTOs, não entities. Erros tipados com HttpException"
  • De exemplos do padrão desejado: "Siga o mesmo padrão deste service existente: [cole código]"
  • Itere, não aceite de primeira: "Bom, mas adicione tratamento de erro para caso o registro não exista. E adicione logging estruturado"

2. Revisando Código Gerado por IA — Checklist Crítico

Checklist de Review de Código Gerado por IA
  1. Segurança: SQL injection? XSS? Secrets hardcoded? Válidação de input? Sanitização de output?
  2. Error Handling: O que acontece quando a API externa falha? Timeout? Retry com backoff? Circuit breaker?
  3. Edge Cases: Null, undefined, string vazia, array vazio, números negativos, datas inválidas, Unicode, emojis?
  4. Performance: N+1 queries? Loop dentro de loop? Objetos grandes em memória? Missing indexes?
  5. Concurrency: Race conditions? Deadlocks? Operações não-atomicas em dados compartilhados?
  6. Testabilidade: Dependências são injetáveis? O código pode ser testado unitariamente sem banco/rede real?
  7. Naming: Nomes fazem sentido no domínio? Abreviações confusas? Consistência com o codebase existente?
  8. Patterns: Segue os patterns do projeto? Ou introduz um padrão novo que conflita com os existentes?
  9. Hallucinations: APIs que não existem? Parametros inventados? Versões de bibliotecas incorretas?
  10. Over-engineering: E mais complexo do que precisa ser? Abstractions desnecessárias? Design patterns forçados?

3. Construindo Features com IA

O senior sabe usar IA como ferramenta, não como muleta:

  • Geração de boilerplate: DTOs, entities, migrations, CRUD básico. IA faz em segundos. Você válida e ajusta
  • Refactoring assistido: "Extraia esse switch/case em Strategy pattern". IA é excelente nisso
  • Testes: "Gere testes unitários para este service cobrindo happy path e error cases". IA gera 80%, você ajusta 20%
  • Documentação: "Gere JSDoc para esta função" ou "Escreva o ADR para esta decisão". IA como ghost writer
  • Debug: "Este erro está acontecendo: [stack trace]. O contexto e: [código]. Qual a causa provável?"

Entendendo Limitações da IA

  • Contexto limitado: A IA não ve todo seu codebase. Ela pode gerar código correto sintaticamente mas que conflita com patterns existentes
  • Hallucinations: A IA inventa APIs, parametros e até bibliotecas que não existem. SEMPRE verifique documentação
  • Bias de treinamento: A IA tende a gerar código "medio" — o padrão mais comum do GitHub. Pode não ser o melhor para seu caso específico
  • Segurança: A IA não pensa em segurança por padrão. Ela pode gerar SQL concatenado, secrets em plain text ou endpoints sem autenticação

Dimensão 4: Excelência em Engenharia

Code Review Checklist

Um bom code review não é "LGTM". E uma ferramenta de ensino, qualidade e alinhamento:

  • Corretude: O código faz o que o ticket pede? Todos os cenários estão cobertos?
  • Design: A abordagem é a mais simples que resolve? Existe alternativa mais limpa?
  • Consistência: Segue os padrões do projeto? Naming, estrutura de pastas, error handling?
  • Testes: Os testes cobrem happy path E edge cases? Os nomes dos testes descrevem o comportamento?
  • Performance: Operações O(n^2) desnecessárias? Queries sem index? Carregamento lazy onde deveria ser eager?
  • Segurança: Input válidado? Output sanitizado? Autorização verificada?
  • Observabilidade: Logs em pontos críticos? Métricas para monitorar a feature? Error tracking?

Observability-First

O senior pensa em observabilidade ANTES de codar. Não depois de um incidente.

  • Logs estruturados (JSON) com correlation ID desde o primeiro request
  • Métricas de negócio (pedidos/minuto) além de métricas técnicas (CPU, memória)
  • Alertas com thresholds significativos (não "CPU > 80%", mas "taxa de erro > 1% por 5 min")
  • Runbooks para cada alerta — "o que verificar quando este alerta dispara"

Security-First

Segurança não é uma feature — e uma propriedade do sistema:

  • Válidação de input em TODA borda (API, webhook, file upload, message queue)
  • Princípio do menor privilégio (least privilege) — cada serviço só acessa o que precisa
  • Secrets em vault (AWS Secrets Manager, HashiCorp Vault) — NUNCA em env vars de CI ou código
  • Dependency scanning automatizado (Dependabot, Snyk) com bloqueio de merge para vulnerabilidades críticas

Cost-Conscious Engineering

O senior entende que cada decisão técnica tem um custo financeiro:

  • Reserved Instances para workloads previsíveis (economia de 40-60%)
  • Rightsizing — uma instância t3.xlarge rodando a 10% de CPU e desperdicio
  • TTLs agressivos em cache e storage de logs — 90% dos logs nunca são lidos após 7 dias
  • Arquitetura cost-aware: Serverless para spiky workloads, containers para sustained

Testing Strategy

O senior define a estratégia de testes, não apenas "cobertura":

  • Unit tests: Lógica de negócio pura. Rápidos, isolados, mockados. Bulk do testing (70%)
  • Integration tests: Camada de dados (repository + DB real via testcontainers). Verifica queries, migrations, constraints (20%)
  • E2E tests: Fluxos críticos do usuário. Poucos mas essenciais. Login, checkout, pagamento (10%)
  • Contract tests: Quando ha APIs entre times. Pact/MSW para garantir compatibilidade
  • Load tests: Antes de cada release major. k6/Gatling para encontrar bottlenecks

Dimensão 5: Liderança Técnica

Framework de Mentoria

Um senior multiplica o time. A formula: ensine a pescar, não de o peixe.

  • Pair programming dirígido: Não assuma o teclado. Faca perguntas. "O que acontece se esse campo for null?" "Tem outra forma de modelar isso?"
  • Code review como ensino: Não apenas "mude X para Y". Explique o por que. Linke para documentação. De contexto histórico. "A gente já tentou essa abordagem no projeto Z é o problema foi..."
  • Delegação progressiva: Comece delegando tasks com solução conhecida (você sabe o caminho). Depois delegue tasks com solução desconhecida (você confia no processo do mentee)
  • 1:1 técnico regular: "O que você aprendeu essa semana?" "Onde você travou?" "Que decisão você tomou que não tem certeza?" Crie espaço seguro para duvidas

Comúnicando para Stakeholders Não-Técnicos

O CTO não quer ouvir "refatoramos o service layer para usar o Strategy pattern com DI". Ele quer ouvir:

  • Impacto no negócio: "Reduzimos o tempo de adicionar novo método de pagamento de 2 semanas para 2 dias"
  • Risco: "Se não fizermos isso agora, o próximo incidente de pagamento vai levar 4h para diagnosticar ao invés de 20min"
  • Custo de oportunidade: "Cada semana que adiamos essa migração custa R$15K em infra desperdicada"
  • Timeline realista: "3 sprints com risco medio. Se priorizarmos, 2 sprints. Podemos paralelisar se alocarmos mais 1 dev"

Influência sem Autoridade

O senior influência decisões sem ter cargo de gerência:

  • Provas técnicas: Prototipe a solução em 2h ao invés de debater em reuniao de 1h. "Fiz um spike e os números são esses"
  • Documentação de decisões: ADRs criam registro. Quando alguém perguntar "por que escolhemos X?", a resposta existe
  • Consistência: Sejá confiávelmente certo. Quando você diz "vai dar problema", o histórico comprova. Credibilidade e construída com tempo
  • Empowers others: Apoie publicamente ideias boas de juniores. "A sugestão do Joao é melhor que a minha porque..."

Cultura de Engenharia

  • Blameless postmortems: "O sistema falhou" não "o Joao errou". Foco em prevenção, não em culpa
  • Documentation culture: Se não está documentado, não existe. READMEs, runbooks, ADRs, onboarding guides
  • Testing culture: "Testes são parte da feature, não um extra". PR sem testes não passa review
  • Continuous improvement: Tech debt sprints regulares. Não acumular até ser insustentável

Dimensão 6: Visao de Negocio

Entendendo o Usuário

O senior não implementa tickets cegamente. Ele pergunta "por que o usuário precisa disso?" é as vezes descobre que a solução real é diferente do que foi pedido.

  • Participe de user research: Assista a sessões de usabilidade. Vejá onde o usuário trava
  • Métricas de produto: Entenda DAU, retention, conversion rate, churn. Essas métricas guiam prioridades
  • Proposta de solução técnica: "O ticket pede um campo de busca. Mas olhando os dados, 80% dos usuários filtram por categoria. Um filtro por categoria resolve melhor e e 3x mais rápido de implementar"

Technical Debt Quadrant

Nem toda tech debt é ruim. Martin Fowler define 4 tipos cruzando deliberada/inadvertida com prudente/imprudente:

Deliberada + Prudente
"Sabemos que não é ideal,
mas precisamos entregar.
Vamos pagar depois."
Inadvertida + Prudente
"Agora que entregamos,
percebemos como deveria
ter sido feito."
Deliberada + Imprudente
"Não temos tempo
para design.
Vai assim mesmo."
Inadvertida + Imprudente
"O que é camada
de serviço?"
(Falta de conhecimento)
  • Deliberada + Prudente — Aceitável. Você sabe o trade-off. Crie ticket de tech debt com deadline
  • Inadvertida + Prudente — Natural. Aprendizado continuo. Refatore quando perceber
  • Deliberada + Imprudente — Perigosa. Corte de esquina consciente sem plano de pagamento. Acumula juros compostos
  • Inadvertida + Imprudente — Crítica. Indica que o time precisa de mentoria e treinamento. Não e debt — e falta de fundação

TCO Analysis

Antes de propor uma mudança técnica, calcule o custo total:

Proposta: Migrar de REST para GraphQL

--- Custos de Migração ---
Engenharia: 2 devs * 3 meses = R$180K
Risco de regressão: 2 semanas de bug fixing = R$30K
Treinamento do time: workshops + ramp-up = R$15K
Downtime parcial durante migração: R$5K
Total migração: ~R$230K

--- Beneficios Anuais ---
Redução de over-fetching: -30% de banda = R$12K/ano
Velocidade de desenvolvimento (frontend): +20% = R$60K/ano
Menós endpoints para manter: -15% ops = R$20K/ano
Total benefício: ~R$92K/ano

--- ROI ---
Break-even: 230K / 92K = ~2.5 anos
Decisão: NÃO fazer agora. REST atende bem. Revisitar quando
volume de dados ou número de clientes mobile justificar.

12 Livros Essenciais para o Dev Senior
  1. Clean Code — Robert C. Martin. A fundação. Nomes, funções, formatação, tratamento de erro
  2. Clean Architecture — Robert C. Martin. Dependency Rule, boundaries, layers. Como organizar sistemas
  3. Designing Data-Intensive Applications (DDIA) — Martin Kleppmann. A biblia de sistemas distribuídos. Replicação, particionamento, consensus

  5. The Pragmatic Programmer — Hunt & Thomas. Sabedoria prática de engenharia. DRY, tracer bullets, ortogonalidade
  6. Fundamentals of Software Architecture — Richards & Ford. Architecture characteristics, styles, decisions. O livro que conecta code a arquitetura
  7. Building Microservices — Sam Newman. Quando (e quando não) usar microsserviços. Decomposicao, comunicação, deploy
  8. Domain-Driven Design — Eric Evans. Bounded contexts, aggregates, ubiquitous language. Essencial para domínios complexos
  9. Accelerate — Forsgren, Humble & Kim. Prova cientifica de que DevOps practices (CI/CD, trunk-based) melhoram resultados. DORA metrics
  10. Staff Engineer — Will Larson. O que vem depois de senior: staff, principal. Como operar no nível de organização
  11. The Manager's Path — Camille Fournier. Mesmo se você não quer ser manager. Entender a hierarquia, expectativas e dinâmicas de poder
  12. Thinking in Systems — Donella Meadows. Pensamento sistemico. Feedback loops, delays, emergent behavior. Aplicável a arquitetura e organizações

Mini System Design: Sua Primeira Semana como Tech Lead

Cenário: Você acabou de ser promovido a Tech Lead de um time de 6 pessoas trabalhando em um e-commerce. O sistema tem um monolito Node.js de 3 anos, 0 testes, deploys manuais é o antigo tech lead saiu. O que você faz na primeira semana?

Dia 1-2: Entender o Estado Atual

  • 1:1 com cada dev: "O que funciona bem? O que te frustra? Onde você vê risco? O que você mudaria se pudesse?" (Ouvir > Falar)
  • 1:1 com Product Manager: "Quais são as prioridades dos próximos 3 meses? Onde a tech está bloqueando o produto?"
  • Audit do codebase: Estrutura de pastas, dependências desatualizadas, padrões (ou falta deles), tamanho dos arquivos, complexidade
  • Audit da infra: Como é o deploy? Quanto demora? Tem CI? Tem staging? Monitoramento? Alertas? On-call?

Dia 3-4: Quick Wins e Plano

  • Quick win 1: Configurar CI básico (GitHub Actions: lint + build). 2 horas de trabalho, impacto imediato
  • Quick win 2: Adicionar health check endpoint + uptime monitor (UptimeRobot). 30 min de trabalho
  • Quick win 3: Documentar o processo de deploy atual. Mesmo que manual, documente cada passo
  • Plano de 30-60-90 dias: 30d = CI/CD + primeiros testes. 60d = Monitoramento + observabilidade. 90d = Primeiro módulo com testes > 60% coverage

Dia 5: Comúnicar e Alinhar

  • Apresentar findings para o time: "Aqui está onde estamos. Aqui está o risco. Aqui está o plano. O que vocês acham?" (Co-criação, não imposicao)
  • Apresentar para gestão: "O risco principal e X. O custo de não enderecar e Y. Precisamos de Z sprints dedicados"
  • Estabelecer agreements: "A partir de agora: PR obrigatório, CI deve passar, testes para código novo (não retroativo)"
  • NÃO fazer: Reescrever tudo, escolher novo framework, criticar decisões anteriores, fazer mudanças grandes sem consenso
✦
Princípio fundamental: Os primeiros 30 dias são para ouvir e entender, não para mudar. Mudanças precipitadas sem contexto destroem credibilidade. Conquiste confiança primeiro mostrando que você entende a realidade — depois proponha mudanças com evidências.

Armadilhas Comuns

❌
Confundir senior com "sabe tudo": O senior diz "não sei, mas vou descobrir" com frequência. A diferença é que ele sabe COMO descobrir rapidamente e tem o framework mental para avaliar o que aprendeu. Fingir saber é o anti-padrão mais perigoso.
⚠
Gold Plating: Gastar 3 semanas refatorando algo que funciona bem porque "não está bonito". Se não tem bug, não tem performance issue e não impede features futuras — deixe. Perfeição e inimiga da entrega.
⚠
Resistir a IA por orgulho: "Eu não precisó de IA para codar". Você também não precisa de IDE com autocomplete. Mas seria ineficiente sem. IA e uma ferramenta de produtividade. Ignorar por orgulho é como recusar o compilador. Aprenda a usa-lá BEM.
✦
O segredo final: O dev senior na era da IA é um multiplicador. Ele não apenas escreve bom código — ele faz o time inteiro escrever bom código. Ele não apenas resolve problemas — ele cria sistemas onde problemas são previsíveis e previníveis. E ele usa IA não como substituto de pensamento, mas como amplificador de julgamento.

Exercícios Práticos

Exercício 1: Um junior no seu time quer usar MongoDB para um novo serviço de pedidos com transações entre pedido, estoque e pagamento. Como você aborda essa conversa?

Resposta: Não diga "não, use PostgreSQL" diretamente. Faca perguntas: "Quais são os requisitos de consistência transacional? O que acontece se o pagamento for aprovado mas o estoque já acabou?" Guie o junior a perceber que transações multi-collection no MongoDB são mais lentas e complexas que transações ACID no PostgreSQL. Pergunte: "Você já trabalhou com transactions multi-document no Mongo? Quais são as limitações?" Se ele não sabe, sugira um spike de 2h comparando as duas abordagens. Deixe ele chegar na conclusão. O aprendizado e 10x mais duradouro quando é auto-descoberto.

Exercício 2: O CTO pede para você estimar quanto custa manter o sistema atual vs reescrever em uma stack moderna. Como você estrutura essa análise?

Resposta: Use TCO Analysis. Custo de manter: (1) Tempo extra por feature devido a tech debt (métricas reais: lead time atual vs ideal). (2) Custo de incidentes (frequência * MTTR * custo/hora da equipe). (3) Custo de oportunidade (features não feitas por estar apagando incendio). Custo de reescrever: (1) Engenharia (devs * meses * salário). (2) Risco de regressions. (3) Feature freeze parcial durante migração. (4) Treinamento na nova stack. Apresente cenários: (A) Manter como está: custo X/ano crescente. (B) Refactoring incremental: investir Y nós próximos 6 meses, reduzir custo para Z/ano. (C) Rewrite: investir W, 12 meses, custo futuro V/ano. Recomende B (incremental) — quase sempre é a opção correta. Rewrites totais falham mais do que funcionam.

Exercício 3: Você usou IA para gerar um service inteiro de 200 linhas. O código compila e os testes passam. O que mais você verifica antes de abrir o PR?

Resposta: Aplique o Checklist de Review completo: (1) Segurança: Inputs válidados? SQL parametrizado? Secrets não hardcoded? (2) Error handling: O que acontece quando dependências falham?
